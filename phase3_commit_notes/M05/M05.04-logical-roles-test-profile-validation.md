# M05.04: Logical Roles And Test Profile Validation

Worktree: `../worktrees/openwrt-bpi-r4-pro-8x`
Test image basis: worktree commit `e1d1d40adf` (`M05.03.c`)
Date: 2026-06-16 local test session

## Scope

This records the M05.04 runtime validation performed after the M05.03 combo mux
implementation.

M05.04 is about final OpenWrt logical roles and defaults at runtime:

```text
combo-lan must behave as a LAN bridge member.
combo-wan must behave as the WAN device.
combo-lan must stay LAN regardless of whether its mux selects RJ45 or SFP.
combo-wan must stay WAN regardless of whether its mux selects RJ45 or SFP.
Temporary M05 names such as m05wan and mxwan must not be used.
```

This record covers the current fixed lab test profile. It does not cover a
fresh-default firstboot run, because that would conflict with the active test
network layout.

## Fixed Test Profile

The router UCI network setup was fixed and committed before testing:

```text
network.lan.device = br-lan
network.lan.proto  = static
network.lan.ipaddr = 192.168.8.1
network.lan.netmask = 255.255.255.0

network.wan.device = combo-wan
network.wan.proto  = static
network.wan.ipaddr = 192.168.9.1
network.wan.netmask = 255.255.255.0

dropbear Interface = lan
dropbear PasswordAuth = on
dropbear RootPasswordAuth = on

network.m05wan deleted
network.mxwan deleted
```

This is a lab profile, not a claim that the image's fresh default WAN protocol
should be static. The point of this profile is to make repeated runtime tests
stable without manually rebuilding the router config each time.

## Physical Setup

Static baseline:

```text
host USB Ethernet -> router lan1 / br-lan / 192.168.8.1 for SSH
host eno1 10G     -> combo-wan RJ45
host eno2 2.5G    -> combo-lan RJ45
combo-lan SFP1    -> removed
combo-wan SFP2    -> removed
```

Host-side namespace naming was inherited from earlier tests and is physically
misleading:

```text
combo-rj45 netns -> physically eno1 / 10G NIC
combo-sfp  netns -> physically eno2 / 2.5G NIC
```

The physical wiring, not the namespace name, was used to interpret results.

## Static Baseline

After applying and committing the UCI profile, the router was rebooted. After
reboot, the committed profile persisted:

```text
br-lan:    192.168.8.1/24
combo-wan: 192.168.9.1/24
```

Bridge membership:

```text
br-lan members:
combo-lan fpc lan1 lan2 lan3 lan4 lan5

combo-lan in br-lan: yes
combo-wan in br-lan: no
```

Baseline link state:

```text
combo-lan:
  Port: Twisted Pair
  PHYAD: 24
  Speed: 2500Mb/s
  Link detected: yes

combo-wan:
  Port: Twisted Pair
  PHYAD: 28
  Speed: 10000Mb/s
  Link detected: yes
```

The speed difference is expected from the host wiring: `eno2` is a 2.5G NIC on
combo-lan RJ45, while `eno1` is a 10G NIC on combo-wan RJ45.

Static data-plane checks:

```text
combo-lan RJ45 / eno2 / 192.168.8.x:
  host -> router ping: OK
  router -> host ping: OK
  host -> router TCP payload: OK
  router -> host TCP payload: OK

combo-wan RJ45 / eno1 / 192.168.9.x:
  host -> router ping: OK
  router -> host ping: OK
  router -> host TCP payload: OK
```

WAN-side SSH was intentionally refused because `dropbear` was bound to LAN.
That refusal was treated as correct policy behavior, not a link failure.

The reboot log showed both muxes starting from SFP-absent and selecting RJ45:

```text
combo-wan mux: MOD_DEF0 SFP present=0, desired channel 0
combo-lan mux: MOD_DEF0 SFP present=0, desired channel 1
combo-wan: Link is Up - 10Gbps/Full
combo-lan: Link is Up - 2.5Gbps/Full
```

## Dynamic Combo-LAN Test

### Step 1: Insert Combo-LAN SFP Module Without Cable

Action:

```text
Insert SFP module into combo-lan SFP1.
Do not connect a cable to the SFP module.
Keep eno2 connected to combo-lan RJ45.
Keep eno1 connected to combo-wan RJ45.
```

Observed:

```text
mxl862xx ... Ethernet mux: MOD_DEF0 SFP present=1, desired channel 0
mxl862xx ... Ethernet mux: switched channel 1 -> 0
sfp sfp1: module XZSNET XZS-SFP10G-T ...
combo-lan: Port FIBRE, Link detected: no, carrier=0
combo-lan remains a br-lan member
```

Result:

```text
combo-lan RJ45 data path stopped, as expected.
combo-wan RJ45 stayed up.
No fallback to RJ45.
No Oops or panic.
```

This proves the no-cable SFP-present case no longer triggers the earlier
`present=1 + link down` panic window.

### Step 2: Move eno1 From Combo-WAN RJ45 To Combo-LAN SFP

Action:

```text
Move eno1 from combo-wan RJ45 to the inserted combo-lan SFP module.
```

Observed:

```text
combo-wan: Link is Down
combo-lan: Link is Up - 10Gbps/Full - flow control off
combo-lan remains in br-lan
```

Result:

```text
combo-lan SFP / eno1 / 192.168.8.x:
  host -> router ping: OK
  router -> host ping: OK
  host -> router TCP payload: OK
  router -> host TCP payload: OK

combo-lan RJ45 / eno2:
  ping: FAIL, expected because the mux selected SFP.
```

### Step 3: Remove Only The Combo-LAN SFP Cable

Action:

```text
Unplug eno1 from the combo-lan SFP module.
Keep the SFP module inserted.
```

Observed:

```text
combo-lan: Link is Down
SFP EEPROM remains readable
No new mux switch
No fallback to RJ45
```

Result:

```text
SFP-present/link-down holds the combo-lan mux on SFP.
combo-lan RJ45 remains inactive.
No Oops or panic.
```

### Step 4: Reconnect The Combo-LAN SFP Cable

Action:

```text
Reconnect eno1 to the already inserted combo-lan SFP module.
```

Observed:

```text
combo-lan: Link is Up - 10Gbps/Full - flow control off
No new mux switch
```

Result:

```text
combo-lan SFP / eno1 / 192.168.8.x:
  host -> router ping: OK
  router -> host ping: OK
  host -> router TCP payload: OK
  router -> host TCP payload: OK
```

### Step 5: Remove Combo-LAN SFP Module And Restore Combo-WAN RJ45

Action:

```text
Unplug eno1 from combo-lan SFP.
Remove the combo-lan SFP module.
Reconnect eno1 to combo-wan RJ45.
```

Observed:

```text
sfp sfp1: module removed
mxl862xx ... Ethernet mux: MOD_DEF0 SFP present=0, desired channel 1
mxl862xx ... Ethernet mux: switched channel 0 -> 1
combo-lan: Link is Up - 2.5Gbps/Full
combo-wan: Link is Up - 10Gbps/Full
```

Result:

```text
combo-lan RJ45 / eno2 / 192.168.8.x:
  host -> router ping: OK
  router -> host ping: OK

combo-wan RJ45 / eno1 / 192.168.9.x:
  host -> router ping: OK
  router -> host ping: OK
```

This completed the combo-lan dynamic role test and returned the router to the
static RJ45 baseline.

## What This Proves

This M05.04 test proves:

1. The fixed lab network profile survives reboot and does not require manual
   reconfiguration for each test cycle.
2. `combo-lan` remains the LAN logical device across RJ45 and SFP mux states.
3. `combo-wan` remains the WAN logical device across RJ45 and SFP mux states.
4. `combo-wan` does not enter `br-lan`.
5. `combo-lan` remains a `br-lan` member even when its selected physical mux
   side has no carrier.
6. Both combo muxes use SFP module presence, not link-up, as the selection
   signal.
7. SFP-present but link-down does not trigger fallback to RJ45.
8. The panic-prone `present=1 + link down` condition was exercised on both
   combo-lan and combo-wan without Oops or kernel panic.
9. `combo-lan` SFP and `combo-wan` SFP both negotiate 10G with the available
   10G host NIC.
10. `combo-wan` RJ45 negotiates 10G with the available 10G host NIC.
11. `combo-lan` RJ45 negotiates 2.5G with the available 2.5G host NIC.

## Not Covered

Fresh-default firstboot config was not tested in this session because replacing
the active UCI test profile would conflict with the current management and
two-subnet lab topology. Fresh-default board.d behavior remains a separate
optional check, not a blocker for this M05.04 runtime test profile.

This test also does not claim line-rate performance, UDP behavior, packet-size
matrix coverage, VLAN bridge correctness, long-duration traffic stability, or
complete error-counter auditing. Those belong to the M05.05 runtime validation
contract.
