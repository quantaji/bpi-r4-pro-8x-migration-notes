## Thermal And Fan Runtime Test

### Test Purpose

The thermal/fan test checked whether the M05 image exposes useful board
temperature telemetry and whether the board fan PWM path can be controlled by
Linux. The relevant hardware conclusion from the schematic review was:

```text
CN36 and CN38 are two physical fan connectors.
Both are connected to the same FAN0_PWMD0 / GPIO57 PWM signal.
They are not independently controlled fan channels.
Tach feedback was not treated as validated input.
```

### Runtime Checks

The runtime checks used the router with the USB management link on `lan1`.
The test inspected the kernel thermal/hwmon sysfs surfaces and exercised the
PWM fan cooling state manually.

The fan was tested at multiple cooling states:

```text
cooling state 0: fan output off
cooling state 2: fan output on at an audible medium level
cooling state 3: fan output on at a louder/high level
```

User-side acoustic/visual observations were used because the board fan tach
feedback was not treated as a reliable Linux input for this step:

```text
state 0: both connected fans stopped
state 2: fans spun up and were audibly running
state 3: higher/louder fan behavior was observed after retest
```

### Temperature Visibility

Runtime temperature visibility was checked through sysfs, not through LuCI as
a blocker. The tested image exposed thermal/hwmon style telemetry for the
platform components that were brought into scope before this closeout:

```text
MT7988 thermal zones / hwmon-style temperature inputs
MxL switch temperature visibility
AS21xxx PHY temperature visibility for the two AS21 instances
SFP DDM temperature when the tested SFP module was inserted
```

The AS21xxx temperature work had an important negative test before the final
runtime check: a vendor-style temperature command path caused AS21 IPC timeout
logs such as `aeon_ipc_wait_cmd failed: -110`. That was not accepted as final
behavior. The final direction was to keep the implementation compatible with
the current OpenWrt/upstream AS21xxx driver API and verify that both AS21 PHY
instances expose separate hwmon temperature inputs without relying on the
vendor debugfs/control surface.

LuCI was not treated as a kernel/runtime blocker. The conclusion was:

```text
sysfs thermal/hwmon visibility is the M05 runtime requirement.
LuCI display would need optional LuCI/statistics integration or a dedicated
status page and is not required to prove the kernel thermal/fan path.
```

### What This Proves

The fan and thermal checks prove:

1. The board fan PWM path is functional from Linux.
2. The two fan connectors behave as a shared-PWM fan output, matching the
   schematic interpretation.
3. The tested image can stop and start the fan from sysfs cooling state
   control.
4. Temperature telemetry is available through kernel sysfs surfaces for M05
   diagnostics.
5. AS21 temperature support must stay aligned with the current OpenWrt driver
   APIs; copying the vendor temperature command path directly is not valid.

## DSA/VLAN Tagged Forwarding Test

### Test Purpose

This test was added after plain bridge/LAN traffic had already worked. The
goal was to prove that tagged VLAN traffic can cross the DSA path between the
USB-managed LAN side and the MxL-backed combo-lan side.

The test specifically checked:

```text
bridge VLAN 100 configuration on lan1 and combo-lan
802.1Q-tagged frames on both physical host links
FDB learning for both endpoints on VLAN 100
bidirectional ICMP
bidirectional TCP payload
bidirectional UDP payload
link-event behavior while VLAN 100 is configured
absence of kernel panic/Oops during combo-lan and combo-wan link changes
```

### Test Topology

Physical setup:

```text
host USB Ethernet -> router lan1 / br-lan / 192.168.8.1
host eno1         -> router combo-wan RJ45
host eno2         -> router combo-lan RJ45
```

Host-side VLAN isolation:

```text
usbv100 on USB Ethernet:             192.168.100.2/24
combo-lan-test netns containing eno2
eno2v100 inside combo-lan-test:      192.168.100.3/24
```

Router-side temporary VLAN setup:

```text
bridge vlan add dev lan1 vid 100
bridge vlan add dev combo-lan vid 100
bridge vlan add dev br-lan self vid 100
br-lan vlan_filtering=1
brlan100 temporary CPU VLAN endpoint: 192.168.100.1/24
```

The temporary `brlan100` CPU endpoint rejected ICMP because of router policy.
That was not treated as a VLAN forwarding failure because the actual target of
the test was port-to-port tagged forwarding between `lan1` and `combo-lan`.

### Static VLAN Baseline

Baseline results:

```text
usbv100 -> eno2v100 ping: OK
eno2v100 -> usbv100 ping: OK
tcpdump on eno2: ethertype 802.1Q, vlan 100 observed
tcpdump on USB Ethernet: ethertype 802.1Q, vlan 100 observed
bridge FDB learned USB MAC on lan1 vlan 100
bridge FDB learned eno2 MAC on combo-lan vlan 100
```

Traffic results:

```text
TCP usbv100 -> eno2v100: about 2.35 Gbit/s, 0 retransmits
TCP eno2v100 -> usbv100: about 2.35 Gbit/s, 0 retransmits
UDP 1G usbv100 -> eno2v100: passed, very small or zero loss
UDP 1G eno2v100 -> usbv100: passed, very small or zero loss
```

Small UDP receive loss around `0.013%` to `0.015%` was observed in some runs.
That was treated as normal UDP test-host measurement noise, not as a link or
driver failure. TCP was clean and ICMP was lossless.

### Interactive Link-Event VLAN Test

The interactive sequence was:

```text
1. Unplug combo-lan RJ45 / eno2.
2. Replug combo-lan RJ45 / eno2.
3. Unplug combo-wan RJ45 / eno1.
4. Replug combo-wan RJ45 / eno1.
5. Unplug combo-lan RJ45 / eno2 again.
6. Replug combo-lan RJ45 / eno2 again.
```

Observed behavior:

```text
combo-lan unplug:
  combo-lan link went down
  br-lan port for combo-lan entered disabled state
  VLAN 100 ping failed as expected
  no panic/Oops/warn

combo-lan replug:
  combo-lan returned to 2.5G/full
  br-lan port returned to forwarding
  VLAN 100 ping restored
  TCP restored at about 2.35 Gbit/s with 0 retransmits

combo-wan unplug:
  combo-wan link went down
  combo-lan VLAN 100 forwarding stayed up
  no panic/Oops/warn

combo-wan replug:
  combo-wan returned to 10G/full
  combo-lan VLAN 100 forwarding stayed up
  no panic/Oops/warn

final combo-lan unplug/replug:
  FDB relearned correctly
  tagged captures on both host physical links still showed VLAN 100
  final TCP and UDP tests passed
```

Final traffic and capture results:

```text
tcpdump on eno2 lower device: VLAN 100 request/reply frames observed
tcpdump on USB lower device: VLAN 100 request/reply frames observed
tcpdump packet drops: 0
final TCP both directions: about 2.35 Gbit/s, 0 retransmits
final UDP 1G usbv100 -> eno2v100: passed, about 0.013% receive loss
final UDP 1G eno2v100 -> usbv100: passed, 0% loss
```

The final error-counter check found cumulative `combo-lan` `ip -s` counters
with `RX errors=2` and `dropped=184`, but those counters did not grow during a
follow-up 20-packet ping run. The MxL-specific discard counters stayed clean:

```text
TxAcmDroppedPkts: 0
RxExtendedVlanDiscardPkts: 0
MtuExceedDiscardPkts: 0
```

After the test, all temporary VLAN and namespace state was cleaned:

```text
router brlan100 removed
VLAN 100 entries removed from lan1, combo-lan, and br-lan self
br-lan vlan_filtering restored to 0
host usbv100 removed
combo-lan-test namespace removed
eno2 returned to the default host namespace and set up
```

### What This Proves

The VLAN test proves:

1. The MxL-backed `combo-lan` DSA path forwards 802.1Q-tagged traffic.
2. `lan1 <-> combo-lan` tagged L2 forwarding works after the DSA/VLAN fix.
3. FDB learning works for VLAN 100 on both sides.
4. Tagged frames are visible on both physical host links, so the test did not
   accidentally use untagged fallback traffic.
5. `combo-lan` link loss disables the expected bridge port and restores cleanly.
6. `combo-wan` link events do not disturb `combo-lan` VLAN forwarding.
7. No kernel panic/Oops was observed during these VLAN and link-event tests.

## LAN1-LAN5 Port Spot Check

### Test Purpose

After the VLAN closeout, `lan1` through `lan5` were tested directly to close the
remaining M05 question about ordinary LAN ports. This was a functional port
spot check, not a long-duration throughput benchmark.

Telemetry for every port:

```text
link state and negotiated speed through ethtool
bridge forwarding/disabled state
FDB learning for the USB Ethernet MAC
host -> router ping
router -> host ping
host -> router TCP payload over SSH
router -> host TCP payload over SSH
ip -s link counters
ethtool -S error/drop/discard counters
dmesg link and error logs
```

The test kept `combo-lan`/`eno2` as the SSH management path so that the USB
test cable could be moved across `lan1` through `lan5`.

Host-side addresses:

```text
USB Ethernet test host: 192.168.8.144/24
combo-lan management host: 192.168.8.115/24
router br-lan: 192.168.8.1/24
```

### Results

| Port | Driver path | Negotiated link | Bridge state while cabled | Result |
| --- | --- | --- | --- | --- |
| `lan1` | MxL862xx | 2.5G/full | forwarding | pass |
| `lan2` | MxL862xx | 2.5G/full | forwarding | pass |
| `lan3` | MxL862xx | 1G/full | forwarding | pass |
| `lan4` | MxL862xx | 2.5G/full | forwarding | pass |
| `lan5` | MT7530/MMIO | 1G/full | forwarding | pass |

For each port:

```text
host -> router ping: OK
router -> host ping: OK
host -> router TCP payload: OK
router -> host TCP payload: OK
FDB learned the USB Ethernet MAC on the currently cabled port
error/drop/discard counters stayed at 0 for the tested port
dmesg showed expected Link Down/Link Up and bridge state transitions only
```

`lan3` and `lan5` negotiated at 1G in this test environment. That was recorded
as an observed link-speed result, not as a functional failure. The functional
data path was clean on both ports.

After the test, the USB cable was returned to `lan1`. Final restored state:

```text
lan1:       up, 2.5G/full, forwarding
combo-lan:  up, 2.5G/full, forwarding
combo-wan:  up, 10G/full
lan2-lan5:  no carrier, disabled as expected when uncabled
```

### What This Proves

The LAN port spot check proves:

1. `lan1` through `lan4` are functional MxL DSA LAN ports.
2. `lan5` is functional through the MT7530/MMIO switch path.
3. Each tested LAN port enters `br-lan` forwarding state when cabled.
4. FDB learning and bidirectional IP traffic work on every LAN port.
5. Moving the LAN cable across ports does not trigger MxL/MT7530 kernel errors.
6. The final runtime topology was restored to the normal lab state.

## Closeout Interpretation

The remaining M05 runtime questions after M05.04 were:

```text
Does the DSA path handle tagged VLAN forwarding?
Do link events during VLAN forwarding remain stable?
Does the board fan/thermal path work well enough for M05?
Do all ordinary LAN ports function at least as bridge members?
```

The answer from these tests is yes for M05 closeout scope.

M05 still does not claim:

```text
fresh-default firstboot network profile validation
long-duration soak testing
line-rate performance matrix across every port and packet size
PPE/HNAT/WED acceleration behavior
Wi-Fi thermal policy integration
LuCI-native temperature display
independent control of the two physical fan connectors
```

Those items are either explicitly waived for this lab environment or belong to
later steps such as M06, M08, M09, or M11.
