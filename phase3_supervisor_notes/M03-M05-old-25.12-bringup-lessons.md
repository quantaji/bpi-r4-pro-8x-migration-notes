## Phase Ownership Summary

### M03

M03 should own only static board-service facts:

- board identity;
- RT5190A and CPU/CCI supply description;
- I2C mux and local I2C devices;
- PCA9555, keys, LEDs, RTC, AT24;
- SPI-NAND `bl2` / `Factory` / `ubi` static partition geometry;
- Factory MAC offset evidence as evidence only.

Accepted decision after checking the current committed worktree: M02 only
implemented SD no-install boot/rootdisk overlay semantics. It did not correct
the SPI-NAND `Factory` partition geometry. Therefore M03 should correct the
static NAND map to preserve `Factory`, while keeping MAC consumers disconnected.

M03 must not claim Ethernet MAC identity is solved. The old repo proved that
the documented Factory MAC window exists as a vendor offset convention, but on
the tested board it reads all `ff`. Therefore M03 may describe the `Factory`
partition, but GMAC MAC consumers must remain gated until M04 chooses a runtime
MAC policy. If M03 adds fixed-layout MAC cells at all, they should be treated as
unconsumed evidence nodes, not as a validated MAC source.

M03 should also carry these old-repo facts:

- GPIO4 is switch reset evidence, not Wi-Fi 12 V evidence, unless newer
  evidence contradicts the old runtime/schematic result.
- `kmod-gpio-pca953x` is needed for PCA9555-backed board services at runtime.
- RT5190A `buck4` / `ldo` voltage divergence was closed in M03 by keeping the
  schematic-backed 1.8 V target values and rejecting the vendor
  `850000` / `1200000` rail mapping.
- Vendor `ina226@40` is not current 8X hardware truth. The 8X schematic marks
  U13 as not populated and runtime probe fails as `ina2xx 3-0040`; M03 should
  omit the enabled DTS node and omit `kmod-hwmon-ina2xx`.

### M04

M04 owns management network defaults and MAC policy:

- final or staged LAN/WAN membership;
- whether current M04 uses conservative bring-up names or waits for M05 netdevs;
- how to handle MAC addresses when Factory cells are invalid;
- whether persistent fallback through U-Boot env is allowed now or must wait for
  the storage/env owner.

Old evidence says the tested board has no valid Ethernet MACs at:

- `Factory + 0xfffee` for GMAC2;
- `Factory + 0xffff4` for GMAC0;
- `Factory + 0xffffa` for GMAC1.

The old runtime also found:

- U8 AT24 `0x57` contains a board-id string like `R4PRO8X-BBC30219`, not MACs;
- vendor image uses random/local Ethernet MACs on this board;
- no UCI `macaddr` was found in vendor runtime;
- non-Pro BPI-R4 style fallback stores generated MACs in U-Boot env.

Design impact: M04 should not use vendor Factory offsets blindly. The likely
target design is an explicit fallback policy. If persistent fallback needs
`fw_printenv` / `fw_setenv`, M04 needs an explicit early-env decision or a
handoff to the storage/env owner.

For board defaults, the old repo converged to:

```text
LAN: lan1 lan2 lan3 lan4 lan5 combo-lan fpc-lan
WAN: combo-wan
```

That naming followed the selected 25.12 driver/DTS organization, not vendor
runtime `mxl_lan*` names. Current M04 should not invent ports before M05 exposes
them. A staged M04 default can be conservative, but final M04/M05 convergence
must be based on actual netdev enumeration.

### M05

M05 owns all wired-runtime issues found by the old repo:

- AS21xxx driver binding, firmware load, C45 feature handling, autoneg, and IPC;
- MxL86252C DSA/tagger/PHY package stack;
- old MxL firmware compatibility;
- GMAC2/MxL switch topology;
- SFP/RJ45 combo runtime switching;
- SFP shared GPIO handling;
- physical link, traffic, DHCP, bridge forwarding, SFP module tests, LEDs, and
  cold/warm boot stability.

The old repo makes one design point non-optional: M05 must support the board's
old MxL firmware. Runtime showed:

```text
mxl862xx mdio-bus:10: switch ready after ~2400-2490ms, firmware 1.0.70 (build 70)
MaxLinear Ethernet MxL86252 PHYs: firmware 0.77 / 1.0.70.70
```

No usable newer MxL firmware blob was found, and the old Banana Pi discussion
did not produce one. Therefore M05 should not require a firmware update. It
must classify behavior by firmware version and keep compatibility paths for
firmware `1.0.70`.

Old MxL firmware requirements observed in the old tree:

- disable or work around old global PCE rules for firmware below `1.0.80`;
- use legacy SFP/PCS API fallback for firmware older than `1.0.84`;
- keep only ports `9` and `13` ungated for old-firmware SerDes use;
- increase command timeout for old firmware because DSA setup hit
  `-ETIMEDOUT`;
- preserve PR/upstream MxL binding shape rather than importing the vendor 6.6
  driver wholesale.

Old AS21xxx requirements observed in the old tree:

- do not "fix" `mtk_open: could not attach PHY: -22` by changing DTS to
  SFP-only or bypassing `phy28`;
- the root issue was AS21xxx/phylink state, not board.d;
- declare the DT model `ethernet-phy-id7500.9432` where the hardware ID reads
  empty and the board evidence is AS21010PB1;
- load firmware before probe-time IPC sync when `firmware-name` is present;
- add model-specific feature reporting so phylink sees speed modes;
- handle C45 read/autoneg/status behavior;
- retry/defer when the firmware loader is busy;
- use vendor-style AS21010PB1 autoneg where needed;
- match AS21xxx IPC replies to the active opcode to avoid stale-status `-110`.

The early cold-boot `combo-wan support ...00006040` / `mtk_open: could not
attach PHY: -22` message remained transient after these fixes. It should be
tracked, but source should not be changed only to clean the boot log unless
physical WAN link or DHCP proves a user-visible failure.

## MxL DTS Translation Lesson

Do not mechanically copy vendor MxL port indexes.

Vendor 24.10 DTS uses:

```text
CPU port reg = 8
combo/ds-mux port reg = 12
```

The vendor driver maps these through:

```c
#define DSA_MXL_PORT(port) ((port) + 1)
```

The PR/upstream-style MxL driver stack selected in the old 25.12 work uses the
logical DSA port indexes directly. The old repo therefore translated the same
hardware topology as:

```text
vendor ports 0..3 -> logical ports 1..4
vendor disabled 4 -> logical port 5
vendor CPU 8      -> logical port 9
vendor combo 12   -> logical port 13
```

This is a binding/driver semantic translation, not a hardware divergence. M05
must either keep this translation if it reuses the PR driver stack, or document
a different translation if the final selected driver changes.

## Cross-Step Design Gates

1. Factory partition geometry and MAC identity are separate.
   - M03 can describe static partition geometry.
   - M04 decides MAC fallback policy.
   - M05 validates GMAC2/MxL side after those netdevs exist.

2. Board defaults must follow exposed netdevs.
   - M04 can set staged management defaults.
   - Final full topology defaults depend on M05 runtime enumeration.

3. MxL firmware update is not a realistic prerequisite.
   - M05 must support firmware `1.0.70`.
   - Do not design a migration path that assumes a newer MxL blob exists.

4. AS21xxx `-22` is not DTS proof.
   - Keep vendor-style AS21xxx topology unless fresh evidence contradicts it.
   - Fix or backport AS21xxx driver behavior in M05.

5. Shared SFP MODDEF0 GPIO matters.
   - The old repo needed a nonexclusive SFP GPIO request to avoid `sfp1 -16`.
   - M05 must account for SFP framework and mux sharing.

6. Cold-start timing must be handled in drivers, not userspace guesses.
   - MxL switch readiness took around 2.4 seconds.
   - AS21xxx firmware loader could stay busy and needed retry/defer behavior.

## Current Actionable Impact

For current Phase 3 planning:

- M03 should keep Factory partition geometry, but must not close MAC identity.
- M04 should be prompted to decide a real MAC fallback policy and whether early
  U-Boot env writes are allowed.
- M05 should treat AS21xxx and MxL old-firmware compatibility as required, not
  optional polish.
- M05 should preserve both combo-port runtime switching and old-firmware port
  gating.
- Any implementation prompt for M05 should require reading the old Phase 9
  notes and commits above before writing driver code.
